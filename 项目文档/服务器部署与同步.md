# 服务器部署与后续同步

## 当前线上结构

- 公网入口：Apache 监听 80，前端构建文件位于 `/var/www/html/`。
- API：Apache 将 `/api` 与 `/uploads` 反向代理到 `127.0.0.1:5050`。
- 后端：`healthdoc.service` 以非登录系统用户 `healthdoc` 运行，代码入口为 `/opt/healthdoc/current/backend`。
- 数据库：openGauss 容器 `healthdoc-gaussdb`，仅绑定 `127.0.0.1:5432`，数据持久化到 `/var/lib/healthdoc/opengauss`。
- 上传文件：`/var/lib/healthdoc/uploads`。
- 公共知识 RAG：`/var/lib/healthdoc/rag/qdrant` 保存 Qdrant Local 索引，`/var/lib/healthdoc/rag/models` 保存 FastEmbed 模型；均由 `healthdoc` 用户持有，不随 release 删除。
- 生产环境变量：`/etc/healthdoc/healthdoc.env`，权限为 root-only，不进入 Git。

本地开发继续使用 `backend/instance/health_system.db`。服务器通过 `DATABASE_URL` 使用 openGauss。首次部署已经把当时的本地 SQLite 数据迁移到服务器；后续代码发布不会再导入 SQLite，也不会覆盖服务器业务数据。

## RAG 生产前置条件

- 用户上传体检报告的业务 OCR 仍调用华为云；RapidOCR 只在显式 `rag sync` 时为无法直接提取文字的已批准公共 PDF 做一次性离线 OCR。
- Ubuntu 服务器需要 OpenCV 的 `libGL.so.1` 运行库。新机器首次同步前执行 `sudo apt-get install -y --no-install-recommends libgl1`，再用生产虚拟环境验证 `python -c "import cv2"`。
- 首次扫描件 OCR 与嵌入冷建库在 4 GB RAM 机器上可能产生明显内存压力；应至少准备约 4 GB 可用 swap。索引完成后内存会回落，普通聊天请求不会再次执行语料 OCR。
- 如果服务器不能访问 Hugging Face，应在可信开发机完成 `rag_sync.py sync`，仅把已校验的 `fast-bge-small-zh-v1.5` 模型缓存安全传入 `/var/lib/healthdoc/rag/models/`。模型文件和传输包不得提交 Git；上传前后必须核对 SHA-256，并用 `local_files_only=True` 验证离线加载。
- 当前生产机已安装 `libgl1`，并以本机已验收模型缓存完成离线建库；持久化 collection 为 `healthdoc_knowledge_9df42456dd29`，包含 7 个来源、664 个片段，隔离来源为 0。

## 日常发布

本地完成开发并确认测试通过后，在项目根目录运行：

~~~powershell
.\scripts\deploy-server.ps1
~~~

脚本会依次执行 `pip check`、后端测试、npm 生产依赖审计、前端测试和生产构建，打包不含 `.env`、SQLite、上传文件、RAG 运行目录和虚拟环境的发布包，通过 SSH 上传并创建时间戳版本。

远端发布器在切换代码前执行以下 RAG 步骤：

1. 创建并授权 `/var/lib/healthdoc/rag/{qdrant,models,cache,huggingface}`。
2. 在 root-only 环境文件中设置 `RAG_ENABLED=1`、运行快照目录及两个持久化路径。
3. 以 `healthdoc` 用户显式运行新 release 的 `scripts/rag_sync.py sync`；应用启动和用户请求不负责联网同步。
4. 同步失败时恢复原环境并继续运行旧 release；同步成功后才原子切换代码、前端和 systemd 服务。
5. 健康检查失败时恢复旧代码、旧前端和旧环境。

首次部署会下载嵌入模型并加载 RapidOCR 内置模型建立索引；若服务器禁止访问模型源，则按上一节离线预置。首次扫描件 OCR 耗时明显长于普通代码发布，后续内容未变化时复用版本化 collection。Qdrant Local 仅允许当前单 Waitress 后端进程访问；将来启用多后端进程前，必须设置 `RAG_QDRANT_URL` 并迁移到独立 Qdrant 服务。

发布不会导入本机 SQLite，也不会把 `rag_demo_01` 至 `rag_demo_05` 或 100 份合成档案写入生产 openGauss；服务器只同步已批准的公共知识。

发布机必须能够通过 SSH 公钥登录。未配置时先执行 `ssh-copy-id ubuntu@111.229.87.94`（Windows 可手工把本机公钥加入服务器 `~/.ssh/authorized_keys`）；脚本不会保存 SSH 或 sudo 密码。

只有在已经单独完成全部测试时，才可使用：

~~~powershell
.\scripts\deploy-server.ps1 -SkipTests
~~~

## 查看状态与日志

~~~bash
sudo systemctl status healthdoc apache2 docker
sudo journalctl -u healthdoc -n 100 --no-pager
sudo docker ps --filter name=healthdoc-gaussdb
sudo docker logs --tail 100 healthdoc-gaussdb
curl -fsS http://127.0.0.1:5050/api/health
curl -fsS http://127.0.0.1/api/health
sudo grep -E '^(RAG_ENABLED|RAG_RUNTIME_PATH|RAG_STORAGE_PATH|RAG_MODEL_CACHE_PATH)=' /etc/healthdoc/healthdoc.env
sudo du -sh /var/lib/healthdoc/rag/*
~~~

功能 smoke 应使用普通测试账号发起一条公开健康科普问题，确认 SSE `done` 中 `retrieval_status=hit` 且 `knowledge_source_count>0`；不要在服务器日志命令中输出问题正文、知识片段或健康指标值。

## 从本地安全连接数据库

数据库端口不直接开放公网。需要使用桌面数据库工具时，先建立 SSH 隧道：

~~~powershell
ssh -L 15432:127.0.0.1:5432 ubuntu@111.229.87.94
~~~

随后让数据库工具连接 `127.0.0.1:15432`，数据库名和用户名均为 `healthdoc`。密码保存在服务器 root-only 配置中，不应写入仓库。

## 备份范围

至少备份以下内容：

- `/var/lib/healthdoc/opengauss`
- `/var/lib/healthdoc/uploads`
- `/var/lib/healthdoc/rag`
- `/etc/healthdoc/healthdoc.env`

数据库运行期间优先使用 openGauss 的 `gs_dump` 做逻辑备份，不要直接复制正在写入的数据目录。恢复演练应在独立目录或测试实例进行。

## HTTPS

当前只有公网 IP 和 HTTP。绑定域名后，应使用 Certbot 或腾讯云证书配置 Apache 443，并把 80 重定向到 HTTPS。健康档案属于敏感数据，正式对外收集真实信息前必须启用 HTTPS。
