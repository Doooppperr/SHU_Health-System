# 数据库规范化说明

> 文档状态：SQLite schema v2 当前规范化说明，已实施并验证。

## 1. 规范化目标

用户、机构、套餐、邀请码、机构图片、指标字典、档案、指标值、亲友关系和评论分别建模。当前关系保持 3NF，并在已识别函数依赖范围内满足 BCNF。

## 2. 第一至第三范式

- 健康指标以 `health_indicators` 多行保存，不按指标动态增加列。
- 机构套餐、邀请码和图片分别建表，不在 `institutions` 中保存重复数组或多值字段。
- 管理机构只通过 `users.managed_institution_id` 引用，不复制机构名称。
- 图片元数据与二进制文件分离，数据库只保存关系和存储定位信息。
- 指标单位、参考范围保存在指标字典，档案指标只保存引用和规范值。
- 评论通过外键引用用户和机构，不重复保存用户名和机构名。
- 档案展示编号 `health{id}` 由整数主键派生，不重复落库，因此不会形成需要同步维护的第二候选键。

`indicator_dicts.aliases` 作为同一标准指标的完整 JSON 别名集合读取，不参与连接和候选键。

## 3. 候选键

| 表 | 候选键 |
|---|---|
| `users` | `username`、非空 `email`、非空 `managed_institution_id` |
| `institutions` | `(name, branch_name)` |
| `packages` | `(institution_id, name)` |
| `institution_invites` | `institution_id`、`code_hash` |
| `institution_images` | `(institution_id, sort_order)`、`storage_key` |
| `indicator_categories` | `name` |
| `indicator_dicts` | `code`、`(category_id, name)` |
| `friend_relations` | `(user_id, friend_user_id)` |
| `health_indicators` | `(record_id, indicator_dict_id)` |

## 4. BCNF 检查

- 用户主键、用户名、非空邮箱和非空管理机构均决定对应用户属性。
- 机构主键或 `(name, branch_name)` 决定机构属性。
- 邀请码的机构候选键和哈希候选键都决定当前邀请码状态。
- 机构图片的 `(institution_id, sort_order)` 决定图片元数据。
- 套餐、指标字典、亲友关系和档案指标继续由既有候选键决定属性。

“机构来源确认后向机构管理员开放”属于跨表授权规则，不通过在档案中复制管理员 ID 实现，因此不会引入传递依赖。

## 5. 数据完整性

当前 SQLite 约束包括：

- 角色限制为 `user/institution_admin/admin`。
- `institution_admin` 必须有管理机构，其他角色不得绑定。
- 同一机构最多一个机构管理员和一个当前邀请码。
- 邀请码状态限制为 `active/used/revoked`。
- 机构图片排序限制为 0–7，机构内排序唯一。
- 机构和套餐通过 `is_active` 软停用。
- 档案状态限制为 `draft/parsed/confirmed`，机构与套餐保持可空。
- 评论评分、套餐价格、指标来源、参考范围和必填文本继续使用检查约束。

邀请码消费、机构停用、管理员撤销、相册重排和跨机构权限由服务端事务补充校验。

## 6. 健康数据与文件边界

- 普通用户、机构管理员、系统管理员使用不同的服务端查询入口。
- 机构管理员查询不复制数据，只对绑定机构的已确认档案执行受限投影。
- 原始报告由鉴权接口读取，不因机构图片公开而暴露整个上传目录。
- AI 会话、摘要、模型回复和机构查看运行日志不进入业务关系表。
- AI 分析结果同样不落库；多档趋势由每次请求从 `health_records` 与 `health_indicators` 确定性计算。
- `ocr_raw_text` 中的待确认附件是外部 OCR 解析过程的原子暂存快照，不是已确认健康事实，也不参与关系查询。用户确认后，标准化指标仍按 `(record_id, indicator_dict_id)` 写入 `health_indicators`；取消时删除暂存文件并恢复旧快照。

因此三工作台不会引入重复档案或角色专用的健康数据副本。

## 7. 物理落库结果

正式数据库已使用 SQLite 安全重建完成 schema v2 升级：

- 目标版本 `PRAGMA user_version=2`。
- 新旧列取交集并对新字段显式回填。
- 新表空建，旧 `logo_url` 数据迁入相册表。
- 升级前已生成时间戳备份，升级后已验证主键、行数、外键和完整性。
- 新版应用拒绝直接启动未迁移的非空旧库，避免半升级。

升级未删除现有业务数据，也未更换 SQLite 数据库引擎。迁移当日快照为 `user_version=2`、`integrity_check=ok`、外键违规 0，并保留 2 个用户、3 家机构、15 个套餐和 12 份档案。本地数据库不受 Git 跟踪，后续业务行数变化不代表 schema 或规范化状态变化。

## 8. 结论

新增邀请码和机构图片独立表、机构管理员唯一绑定以及健康数据受限投影后，当前模型继续满足 3NF，并在当前业务函数依赖范围内满足 BCNF。跨角色权限由外键、唯一约束、检查约束和服务端事务共同实现。
