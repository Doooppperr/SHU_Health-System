# 数据库规范化说明

## 1. 规范化目标

项目将用户、机构、套餐、指标字典、档案、指标值、亲友关系和评论拆分为独立实体，整体达到第三范式，关键关系通过候选键约束进一步强化。

## 2. 第一范式

- 每个字段保存单一语义。
- 健康指标不以多个动态列存放，而是拆为 `health_indicators` 多行。
- 机构套餐、评论和亲友关系分别独立建表。

`indicator_dicts.aliases` 使用 JSON 保存同一指标的 OCR 别名集合，该字段作为完整值读取和更新，不参与关系连接和候选键。

## 3. 第二范式

所有业务表使用单列主键。依赖于组合业务键的属性都依赖整个业务键：

- 套餐依赖 `(institution_id, name)`。
- 档案指标依赖 `(record_id, indicator_dict_id)`。
- 亲友关系依赖 `(user_id, friend_user_id)`。
- 指标名称依赖 `(category_id, name)`。

## 4. 第三范式

- 机构信息只存于 `institutions`。
- 套餐通过 `institution_id` 引用机构。
- 指标分类只存于 `indicator_categories`。
- 指标定义通过 `category_id` 引用分类。
- 档案指标通过 `indicator_dict_id` 引用单位和参考范围。
- 评论通过外键引用用户和机构，不重复保存用户名或机构名。

这些设计避免了插入、更新和删除异常。

## 5. 候选键

| 表 | 候选键 |
|---|---|
| `users` | `username`、`email` |
| `institutions` | `(name, branch_name)` |
| `packages` | `(institution_id, name)` |
| `indicator_categories` | `name` |
| `indicator_dicts` | `code`、`(category_id, name)` |
| `friend_relations` | `(user_id, friend_user_id)` |
| `health_indicators` | `(record_id, indicator_dict_id)` |

## 6. BCNF 检查

在当前业务定义的函数依赖下，各表的非平凡决定因素均为候选键：

- `users`：`id`、`username` 或非空 `email` 决定用户属性。
- `institutions`：`id` 或 `(name, branch_name)` 决定机构属性。
- `packages`：`id` 或 `(institution_id, name)` 决定套餐属性。
- `indicator_dicts`：`id`、`code` 或 `(category_id, name)` 决定指标定义。
- `friend_relations`：`id` 或 `(user_id, friend_user_id)` 决定关系属性。
- `health_indicators`：`id` 或 `(record_id, indicator_dict_id)` 决定指标值属性。
- 其他实体表由主键决定其余非键属性。

因此不存在“非候选键决定其他属性”的已知函数依赖，当前关系模式在已建模依赖范围内满足 BCNF。跨表业务规则属于参照完整性或业务约束，不通过反规范化冗余字段实现。

## 7. 数据完整性

SQLite 中实际创建并验证的模型约束包括：

- 评分限制为 1–5。
- 套餐价格不得为负数。
- 角色限制为 `user/admin`。
- 档案状态限制为 `draft/parsed/confirmed`。
- 指标来源限制为 `manual/ocr`。
- 参考下限不得高于参考上限。
- 用户不能与自己建立亲友关系。
- 必填文本去除空格后不能为空。

跨表规则由后端集中校验并通过测试覆盖。

## 8. 物理落库状态

规范化不只停留在文档和 ORM 模型。当前本地
`backend/instance/health_system.db` 已按最新模型完成重建：

- 22 个命名 CHECK 约束已写入真实建表 DDL。
- 3 个新增候选键联合唯一约束已写入真实建表 DDL。
- 其他单列和联合唯一约束继续保留。
- 外键检查在每个应用连接上强制开启。
- 旧数据迁移前通过违规扫描，迁移后通过外键和完整性检查。

升级旧 SQLite 文件使用：

```powershell
Set-Location .\backend
.\.venv\Scripts\python.exe .\scripts\upgrade_local_database.py
```

由于 SQLite 不支持直接追加多数表约束，该脚本采用安全重建并自动备份，不是删除业务数据重新初始化。

## 9. 范式优化结论

改回本地 SQLite 只改变数据库运行位置，不改变逻辑模型、范式、候选键或完整性规则。项目关系模式继续满足 3NF，并在当前已识别函数依赖范围内满足 BCNF。
