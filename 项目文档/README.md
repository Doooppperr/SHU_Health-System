# 康康健健 HealthDoc 项目文档索引

> 最后核对：2026-07-21。文档已完成 1.0—3.0 融合整理；当前实现为 3.0/SQLite schema v8，包含机构主体/分院、跨院归档只读与审计、健康时间线/体检数据/健康趋势、套餐版本、最多五人预约组、空位提醒、私有附件和领域约束 AI/OCR。

本目录按“业务主题”而不是“某一轮重构”组织。每份开发文档都说明与其主题相关的 1.0 基础、2.0 流程化和 3.0 平台化演进，并以 3.0/schema v8 作为唯一当前实现基线。原 2.0 临时计划、3.0 冻结方案和 3.0 实施说明已归并到对应主题文档，不再单独维护。

## 推荐阅读顺序

1. 新成员先读根目录 [`README.md`](../README.md)，完成安装、启动和最小验证。
2. 了解完整产品边界时读 [`项目需求与技术方案.md`](项目需求与技术方案.md)。
3. 本地安装、演示、备份或排障时读 [`本地运行与演示指南.md`](本地运行与演示指南.md)。
4. 修改表、约束或迁移脚本前读 [`数据库设计说明.md`](数据库设计说明.md) 与 [`数据库规范化说明.md`](数据库规范化说明.md)。
5. 修改 AI、SSE、OCR 或机构报告状态机前读 [`AI与OCR开发说明.md`](AI与OCR开发说明.md)。
6. 查看当前验证结果时读 [`测试报告.md`](测试报告.md)。
7. 登录合成账号或验收丰富时间线时读 [`测试账号与演示数据.md`](测试账号与演示数据.md)。
8. 服务器发布、状态检查、SSH 隧道或备份时读 [`服务器部署与同步.md`](服务器部署与同步.md)。

## 单一事实来源

- 当前代码行为以路由、模型、服务和测试为最终依据；文档必须与它们同步。
- 当前数据库结构以模型、`schema.py`、`upgrade_local_database.py` 和《数据库设计说明》为准。
- 当前 AI/OCR 协议以服务端路由、前端 API 模块和《AI 与 OCR 开发说明》为准。
- 当前测试数量以《测试报告》顶部为准。
- 当前启动命令以根 README 和《本地运行与演示指南》为准。
- Git 记录代码、文档和一份明确标注的合成演示数据库快照。除《测试账号与演示数据》中的专用合成账号外，真实密码、邀请码明文、API Key、真实健康数据、时间戳数据库备份、RAG 运行数据和上传文件不得写入文档或提交。

## 融合式文档规则

- 每份开发文档先用简短演进表说明该主题从 1.0 到 3.0 解决了什么，再完整描述 3.0 当前契约。
- 历史内容只保留设计原因、兼容关系和迁移意义，不保留已经失效的操作步骤、接口或数据规模。
- 项目中仍存在的能力必须完整记录，即使它不是最近一次重构新增内容。
- 需求、数据库、前端、后端、运行、部署和测试对同一术语必须一致：机构主体、分院、体检数据、自主测量、同行预约、空位提醒和归档报告。
- 新一轮改造完成后直接回填主题文档，不再长期保留“临时计划”“实施说明”一类重复文件。

## 修改与发布检查

- 产品或角色变化：同步需求方案、前后端 README、路由/接口测试和测试报告。
- 数据模型变化：同步模型、schema 版本、迁移脚本、数据库两份说明、备份恢复和测试。
- AI/OCR 变化：同步协议、前后端测试、Mock 行为、错误码和演示步骤。
- 删除能力：同步删除代码、页面、API、测试与过时说明，并确认数据处置。
- 构建或依赖变化：同步前端 README、锁文件、测试报告和 CI。
- 发布前至少执行 Pytest、Vitest、Vite build、依赖检查、数据库完整性/外键检查和 `git diff --check`。
- 发布前必须先备份服务器数据库，再使用当前 schema 与合成快照执行受控数据同步；不得仅依靠 `create_all()` 修改已有 openGauss 表。

## 维护约定

- 数据模型变化不能只调用 `db.create_all()`；SQLite 与生产数据库都必须有明确迁移路径。
- 修改正式数据前必须停止写入并备份数据库；数据库、上传目录和环境配置应组成同一恢复集合。
- 新配置先写 `.env.example`，真实值只写本机 `.env` 或服务器 root-only 环境文件。
- AI/OCR 请求、SSE 事件、报告状态机或错误码变化时，同步专项说明、前后端测试和 Mock。
- 亲友、机构账号、管理员与 AI 的健康数据 DTO 必须分别审查，不能通过复用“完整对象序列化”扩大字段。
- 文档写“当前功能”时必须检查整个项目中所有仍有效内容，不能只记录最近一个重构计划。
