# 本地运行与演示指南

本指南适用于当前已完成的“公开门户 + 三角色工作台 + AI 流式分析 + OCR 两阶段录入 + SQLite schema v2”版本。

## 1. 环境要求

- Windows 10/11、PowerShell 5.1+
- Python 3.10+
- Node.js 20+
- npm 10+
- 无需安装 MySQL、PostgreSQL 或其他数据库服务

## 2. 首次准备

在项目根目录执行：

~~~powershell
Set-Location .\backend
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt

Set-Location ..\frontend
npm ci

Set-Location ..
if (-not (Test-Path .\backend\.env)) {
    Copy-Item .\backend\.env.example .\backend\.env
}
~~~

默认 SQLite 文件：

~~~text
backend/instance/health_system.db
~~~

`backend/instance/*.db` 不受 Git 跟踪，因此 fresh clone 默认没有数据库文件；首次启动后端会创建 schema v2 并写入种子数据。若本机已有数据库才执行下一节检查；换入旧版非空数据库时，后端会拒绝半升级并提示执行迁移脚本。

## 3. 数据库检查与迁移

本机数据库文件已存在时，可在启动前执行只读检查：

~~~powershell
Set-Location .\backend
if (Test-Path .\instance\health_system.db) {
    .\.venv\Scripts\python.exe .\scripts\upgrade_local_database.py --check-only
} else {
    Write-Host "数据库尚未创建；首次启动后端将直接创建 schema v2。"
}
Set-Location ..
~~~

2026-07-11 的迁移验收快照为：

| 项目 | 值 |
|---|---:|
| PRAGMA user_version | 2 |
| users | 2 |
| institutions | 3 |
| packages | 15 |
| health_records | 12 |

本地数据库文件已被 Git 忽略，当前业务行数会随演示变化；上表不是新克隆仓库的固定数据。日常只需确认 `user_version=2`、外键无违规且 `integrity_check=ok`。

旧库升级命令：

~~~powershell
Set-Location .\backend
.\.venv\Scripts\python.exe .\scripts\upgrade_local_database.py
Set-Location ..
~~~

脚本会先在 backend/instance/ 生成 health_system.before-schema-v2-时间戳.db，验证临时库后再原子替换正式文件。升级失败不会覆盖正式数据库。

## 4. 开发模式

### 本地 RAG 与 100 份合成档案

首次准备好依赖后，在 `backend` 目录显式同步白名单语料：

~~~powershell
.\.venv\Scripts\python.exe .\scripts\rag_sync.py sync
.\.venv\Scripts\python.exe .\scripts\evaluate_rag.py
~~~

成功后在 `backend/.env` 设置 `RAG_ENABLED=1` 并重启后端。默认索引、模型、PDF 快照、隔离文件和 source state 都在 `backend/instance/rag/`，不会提交。应用启动与聊天请求不会下载语料或模型。

合成数据脚本默认 dry-run；写入和验收命令如下：

~~~powershell
$env:RAG_DEMO_PASSWORD="至少12字符且仅供本机使用的密码"
.\.venv\Scripts\python.exe .\scripts\seed_rag_demo.py apply
.\.venv\Scripts\python.exe .\scripts\seed_rag_demo.py verify
~~~

验收结果必须是 5 个账号、100 份 `confirmed` 档案、900 条指标值。再次 `apply` 不会新增；`cleanup` 只删除运行时 manifest 记录的合成用户、档案、指标和亲友关系，不影响既有数据。

在项目根目录执行：

~~~powershell
.\scripts\start-full-dev.ps1
~~~

访问：

- 公开门户：http://127.0.0.1:5173/
- 后端健康检查：http://127.0.0.1:5050/api/health

脚本会在新 PowerShell 窗口启动 Flask，并在当前窗口启动 Vite。前端 /api 与 /uploads 请求代理到后端。

需要更换后端端口：

~~~powershell
.\scripts\start-full-dev.ps1 -BackendPort 5051
~~~

## 8. 推荐完整演示顺序

### 8.1 公开门户与普通用户

1. 打开 /，展示项目介绍、核心能力、使用流程、隐私提示和“关于我们”。
2. 点击顶部登录/注册按钮，展示独立品牌页。
3. 在 /register 选择普通用户并注册，登录后确认自动进入 /dashboard。
4. 手工创建健康档案，机构和套餐选择“暂不选取”，确认列表显示 `health+数字` 编号，再录入指标并查看趋势。删除档案不会重排其余现存编号。
5. 再创建一份关联机构的档案，向演示者说明：只有档案 confirmed 后，标准化指标才会只读开放给该机构管理员。
6. 使用 OCR Mock 直接上传图片或 PDF，确认候选映射后创建新档案。
7. 再手工创建一份档案，从列表点击“OCR录入”：验证上传后原状态、指标和正式报告不变；刷新页面可恢复候选，点击“放弃暂存报告”不影响原档案。重新上传后确认页默认“以本报告替换旧 OCR 指标”，确认它会清理新报告未包含的旧 OCR 指标但保留手工指标；再演示切换“合并更新”可保留旧 OCR 指标。
8. 演示亲友添加、授权、代传和趋势。
9. 打开健康 AI，先发送普通问题并确认初始界面没有档案选择器；再询问“我的报告有什么异常”触发内联选档，或点击输入区“引用档案”。每次发送档案前确认隐私提示。
10. 回到档案列表，单选后点击“智能分析（1）”查看全部指标；再选择同一人的多份档案查看服务端计算的趋势。清空选择后可切换另一位所有者。
11. 拖动 AI 面板宽度，确认主页面整体缩放而不是挤压导航；缩小窗口后确认切换为遮罩对话框且键盘焦点进入面板。
12. 关闭 AI 后开启“关怀模式”，确认文字、控件和间距整体放大且顶部导航保持单行；再次打开 AI，确认主页面与侧栏边界衔接且没有横向滚动；缩窄窗口后确认自动回退到原响应式布局。
13. 浏览机构列表，确认卡片展示机构相册第一张封面；进入详情页确认同一封面可见，再浏览套餐并提交评论。对无相册机构应显示“暂无封面”占位而不是破图。

### 8.2 系统管理员

1. 使用当前环境配置的系统管理员账号登录（纯开发库首次可用 admin / admin123），确认进入 /admin/dashboard，且不显示普通用户的亲友、个人健康 AI 菜单。
2. 在机构管理中新增或编辑机构、套餐，上传机构相册并拖拽排序。
3. 验证相册第一张为封面；格式仅 JPEG、PNG、WebP，单张最多 5 MB，每机构最多 8 张。切换到普通用户后确认机构列表和详情页同步展示该封面。
4. 在邀请码管理中选择尚无管理员的机构并签发邀请码。
5. 立即复制明文邀请码用于演示；离开签发结果后系统不会再次显示明文。
6. 在用户管理中查看 user、institution_admin、admin 角色和机构绑定。
7. 展示全局档案监管及评论审核。
8. 演示机构软停用/恢复；说明历史档案、套餐、评论和图片不会因停用丢失。

### 8.3 机构管理员

1. 退出 admin，在 /register 选择机构工作人员，填写刚签发的邀请码并注册。
2. 登录后确认自动进入 /org/dashboard，不能进入 /dashboard 或 /admin/dashboard。
3. 编辑所属机构资料，管理本机构套餐和相册。
4. 查看本机构健康档案与指标趋势：只出现来源为本机构且状态为 confirmed 的档案。
5. 确认页面不显示邮箱、手机号、上传人和原始报告，也不提供修改、补录或删除档案按钮。
6. 回到 admin 用户管理，撤销该机构管理员。
7. 重新刷新该账号，确认其立即失去 /org 权限并降级到普通用户工作台；个人数据仍保留。

## 9. OCR 模式

离线演示：

~~~env
OCR_PROVIDER=huawei
OCR_USE_MOCK=1
~~~

真实 OCR：

~~~env
OCR_USE_MOCK=0
HUAWEI_OCR_ENDPOINT=
HUAWEI_OCR_AK=
HUAWEI_OCR_SK=
HUAWEI_PROJECT_ID=
OCR_PDF_MAX_PAGES=8
OCR_AUTO_CONFIRM_MIN_SCORE=0.92
~~~

真实凭据只能写入 backend/.env，不得放入截图、文档或 Git。

OCR 新建档案上传返回 201，状态为 `parsed`，仍需确认后才成为 `confirmed`；给既有档案补传返回 200，并提供待确认 `attachment_id`。后者的补传、确认、取消以及档案删除使用版本校验；出现 409 时必须重新加载当前档案/待确认结果，不能继续使用旧页面的候选。

真实华为模式的当前解析器显示为 `region-v2`。演示时重点检查：多个表格区域不会混行；表格与非表格文本同时产生候选；低置信度、冲突或非法值行默认标记“需复核”且不勾选；姓名、日期等报告信息与字典外项目显示在“未作为指标/报告信息字段”。不要为了让页面看起来“全匹配”而把未知医疗项目手工映射到含义相近但不同的指标，正确做法是扩展指标字典或保留待人工处理。

给既有档案确认时，前端显式发送：

- `replace_ocr`（默认）：写入本次确认结果，只删除本次未包含的旧 OCR 来源指标，手工来源指标不受影响。
- `merge`：仅 upsert 本次确认结果，保留其他旧 OCR 指标，适合补充报告。

服务端对没有 `ocr_update_mode` 的旧客户端仍默认 `merge`。真实服务 smoke 应只使用合成报告，并且只记录布局、已配置指标数量和结果状态，不记录凭据、云端完整响应或报告健康明细。

## 13. 常见问题

### 后端提示 schema upgrade required

这是旧非空数据库保护机制。停止后端，依次执行：

~~~powershell
Set-Location .\backend
.\.venv\Scripts\python.exe .\scripts\upgrade_local_database.py --check-only
.\.venv\Scripts\python.exe .\scripts\upgrade_local_database.py
~~~

### 前端接口请求失败

~~~powershell
Invoke-RestMethod http://127.0.0.1:5050/api/health
~~~

预期：

~~~json
{"status":"ok"}
~~~

若后端改为 5051，应通过完整启动脚本传入 -BackendPort 5051，让 Vite 代理同步更新。

### 邀请码无法使用

确认机构处于启用状态、尚未绑定机构管理员，并使用最新签发的明文邀请码。邀请码只能成功消费一次；重新签发后旧码立即失效。

### 机构管理员看不到某份档案

必须同时满足：档案显式关联该机构、档案状态为 confirmed、该账号仍绑定并管理该机构。未关联机构的档案不会显示。

### 原始报告公共链接返回 404 或 403

这是预期保护。报告必须通过已登录且有权访问的 /api/records/{id}/file 获取；公共 /uploads 只提供数据库登记的机构图片，未登记文件也会返回 404。

### AI 提示服务未配置

检查 backend/.env 是否有非空 DEEPSEEK_API_KEY，保存后重启后端。离线演示可设置 AI_USE_MOCK=1。

### AI 请求返回 500，Waitress 提示 Connection header

这是旧版流响应错误。当前代码不会设置 WSGI 禁止的 `Connection` hop-by-hop 头。确认本地代码已更新后，彻底停止旧 Waitress 进程并重新执行 `.\scripts\start-full-prod.ps1`；仅刷新浏览器不会替换后端进程。

### OCR 返回 409

- `OCR_ATTACHMENT_CONFLICT`：解析期间档案已变化，重新上传。
- `OCR_ATTACHMENT_STALE`：确认或取消使用了旧附件版本，刷新 OCR 页面恢复最新候选。
- `RECORD_DELETE_CONFLICT`：删除期间附件发生变化，刷新档案列表后重试。

这些冲突保护原报告、指标和暂存文件，不应通过手工修改请求中的 `attachment_id` 绕过。

### Waitress 提示必须设置 JWT_SECRET_KEY

若错误包含：

~~~text
Production startup requires an explicit JWT_SECRET_KEY of at least 32 characters
~~~

处理方式：

1. 关闭报错的后端窗口；若 Vite Preview 仍在运行，在原窗口按 `Ctrl+C` 停止。
2. 回到项目根目录，不要直接执行 `python -m waitress`。
3. 重新运行 `.\scripts\start-full-prod.ps1`。
4. 首次启动出现 `Generated a local production JWT secret in backend/.env.` 属于正常提示，密钥值不会输出。

若必须直接调用 Waitress，则先在 `backend/.env` 手工设置至少 32 字符的 `JWT_SECRET_KEY` 和至少 12 字符且不是 `admin123` 的 `DEFAULT_ADMIN_PASSWORD`。
