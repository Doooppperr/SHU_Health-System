# 数据库设计说明

> 文档状态：SQLite schema v2 当前物理设计；2026-07-13 AI/OCR 优化未新增表或列。

## 1. 运行方式

项目继续使用本地 SQLite：

```text
backend/instance/health_system.db
```

所有连接启用 `PRAGMA foreign_keys=ON` 和 5 秒锁等待。`PRAGMA user_version=2` 标识本次三角色重构后的目标结构。

## 2. 当前表结构

| 表 | 用途 | 关键约束 |
|---|---|---|
| `users` | 三类账号 | 用户名/邮箱唯一，角色检查，管理机构唯一 |
| `friend_relations` | 普通用户亲友授权 | 用户对唯一，禁止自己添加自己 |
| `institutions` | 体检机构 | 名称与分院联合唯一，软停用 |
| `packages` | 机构套餐 | 机构与套餐名联合唯一，价格非负，软停用 |
| `institution_invites` | 每机构当前邀请码 | 机构唯一、哈希唯一、状态检查 |
| `institution_images` | 机构相册 | 机构与排序联合唯一，排序为 0–7 |
| `indicator_categories` | 指标分类 | 分类名唯一 |
| `indicator_dicts` | 指标字典 | 编码唯一，分类与名称联合唯一 |
| `health_records` | 健康档案 | 归属人、上传人、可选来源、状态检查 |
| `health_indicators` | 标准化指标值 | 档案与指标联合唯一 |
| `comments` | 机构评论 | 评分范围 1–5 |

AI 会话、分析结果和机构健康数据访问日志不建业务表；前两者只存在当前浏览器标签页，后者只写不含健康值的后端运行日志。

## 3. 当前关系

```text
institutions 1 ── 0..1 users（managed_institution）
institutions 1 ── 0..1 institution_invites
institutions 1 ── N institution_images
institutions 1 ── N packages
institutions 1 ── N health_records
institutions 1 ── N comments

users 1 ── N health_records（owner/uploader）
users 1 ── N comments / friend_relations
health_records 1 ── N health_indicators
indicator_dicts 1 ── N health_indicators
```

## 4. schema v2 关键字段

### `users`

- `role`：`user`、`institution_admin` 或 `admin`。
- `managed_institution_id`：机构管理员绑定机构；机构内唯一，可空。
- 检查约束保证机构管理员必须绑定机构，其他角色不能绑定。

### `institutions` 与 `packages`

- `is_active`：软停用状态；历史档案仍保留关系。
- 机构停用不会级联物理删除套餐、图片、评论或档案。

### `institution_invites`

- `institution_id`：每机构唯一。
- `code_hash`：高熵邀请码哈希，明文不落库。
- `status`：`active/used/revoked`。
- `issued_by_admin_id`、`used_by_user_id`、`issued_at`、`used_at`、`revoked_at`。

### `institution_images`

- `institution_id`、`storage_key`、`image_url`、`sort_order`、`created_at`。
- `(institution_id, sort_order)` 唯一，`sort_order between 0 and 7`，从而限制每机构最多 8 张。
- 最小排序位置即封面，不额外保存可能漂移的封面标志。

### `health_records`

- `institution_id`、`package_id` 保持可空。
- 机构管理员只读范围为：绑定机构、`status=confirmed`。
- 面向机构管理员的 DTO 不包含上传人、联系方式和 `report_file_url`。
- `display_id=health{id}` 是模型属性/DTO 派生值，不是数据库列或新主键。URL、外键和查询继续使用整数 `id`；删除不会重排其他现存主键，编号可能不连续。
- `ocr_raw_text` 保存 OCR 原始解析快照。给既有档案补传时，快照内临时包含 `_pending_attachment`：不透明 `attachment_id`、待确认报告 URL、上传人和补传前快照。该数据只用于两阶段确认和并发控制，不作为已确认健康指标。
- `ocr_pending_confirmation` 同样是从快照派生的响应布尔值，不是数据库列。

## 5. 完整性与事务规则

- 一机构最多一个管理员；一管理员最多绑定一个机构。
- 每机构最多一个当前邀请码；消费邀请码、创建用户和绑定机构必须原子提交。
- 套餐必须属于档案选择的机构；清空机构同步清空套餐。
- 机构/套餐删除使用软停用，不破坏历史来源。
- 相册重排必须一次提交全部当前图片 ID，归一化为连续的 `0..n-1`。
- 角色撤销后服务端逐请求查库立即生效。
- 图片保存失败或数据库提交失败时清理新文件；删除图片后清理对应存储文件。
- OCR 补传通过 `ocr_raw_text` 快照条件更新：上传只暂存，确认成功才替换正式报告并 upsert `health_indicators`，取消则恢复补传前快照。
- OCR 重传、确认、取消和档案删除使用同一快照 CAS。并发冲突回滚数据库并清理未采用文件，删除成功后同时清理正式和待确认报告。

## 6. 文件访问边界

- 新上传机构图片存入 `uploads/institutions/`；公开路由还会核对 `institution_images.storage_key`，只服务数据库登记的机构图片，并兼容迁移后的旧 logo 子路径。
- `uploads/reports/`：健康报告，仅通过鉴权档案文件接口读取。
- 数据库保存相对 URL/存储键，不保存图片二进制。
- 机构管理员健康接口不返回原始报告地址。

## 7. schema v2 迁移机制

现有 SQLite 引擎继续使用，只迁移表结构：

1. 识别旧版本、表和列，拒绝未知版本。
2. 创建最新结构的临时数据库。
3. 复制新旧交集列并显式回填 `is_active=1`、`managed_institution_id=NULL`。
4. 允许旧库没有邀请码和相册表；将非空旧 `logo_url` 转为排序第一张图片。
5. 设置 `PRAGMA user_version=2`。
6. 校验主键集合、行数、外键、约束和 `integrity_check`。
7. 备份原库并原子替换；失败时原库不变。

新安装创建最新空库并写入版本。非空旧库在迁移前不得由 `db.create_all()` 部分改造。
