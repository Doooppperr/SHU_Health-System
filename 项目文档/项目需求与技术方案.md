# 体检评价与健康档案系统：项目需求与技术方案

> 文档状态：三角色重构与 AI/OCR 优化均已实施，当前版本说明（SQLite schema v2，2026-07-13）。

## 1. 项目目标

构建一个在个人电脑上直接运行的 B/S 健康管理平台。系统继续使用 Vue 3、Flask 与本地 SQLite，以健康档案、指标标准化和长期趋势跟踪为核心，机构服务作为独立功能存在。

## 2. 页面体系与角色

系统采用“公开门户 + 双体系 + 三工作台”。

| 角色 | 默认入口 | 页面体系 | 核心权限 |
|---|---|---|---|
| 访客 | `/` | 公开门户 | 查看项目介绍、关于我们、登录注册和公开 AI 导览 |
| 普通用户 `user` | `/dashboard` | 个人健康中心 | 档案、OCR、指标、趋势、亲友授权、机构服务、评论和健康 AI |
| 机构管理员 `institution_admin` | `/org/dashboard` | 机构运营后台 | 本机构资料、相册、套餐及本机构来源健康数据只读查询 |
| 系统管理员 `admin` | `/admin/dashboard` | 系统管理后台 | 机构、邀请码、用户角色、机构管理员、档案监管和评论审核 |

两类管理员只进入专属后台，不使用普通用户的个人档案、亲友和带档案上下文的健康 AI。前端导航、路由和后端接口同时执行角色隔离，不能仅依赖隐藏按钮。

## 3. 功能需求

### 3.1 公开门户与认证

- `/` 展示项目介绍、核心能力、使用流程、隐私提示和“关于我们”。
- 顶部固定登录、注册入口；登录用户显示与角色匹配的“进入工作台”。
- 登录和注册使用独立品牌化页面，保留图片验证码、JWT Access/Refresh Token。
- 注册分为普通用户和机构工作人员；工作人员必须提交有效邀请码。
- 登录成功后按角色进入 `/dashboard`、`/org/dashboard` 或 `/admin/dashboard`。

### 3.2 普通用户健康中心

- 健康总览显示档案总数、已确认档案数、最近体检、异常指标和最近档案。
- 用户管理本人及已授权亲友的档案、指标和趋势。
- OCR 既可直接创建新档案，也可从现有档案进入；现有档案的解析结果必须先暂存，确认前不得覆盖原报告、状态和指标。
- 用户浏览机构、套餐和公开评论；机构不再是默认首屏。
- 档案列表允许选择同一归属人的任意数量可分析档案，点击“智能分析（N）”后在 AI 侧栏查看结果；单档分析全部指标，多档同时分析指标和趋势。
- 界面档案编号统一为 `health+数字`，内部 API 和数据库仍使用数字主键；删除档案不重排其余现存档案。

### 3.3 档案来源与标准化

- `institution_id`、`package_id` 为可选来源，界面提供“暂不选取”。
- 清空机构时同步清空套餐；仅选择套餐时由服务端推导机构。
- 选择机构或套餐时告知用户：档案确认后，标准化指标会自动向对应机构管理员只读开放。
- 未关联机构的档案仍可维护指标、查看趋势和使用健康 AI。
- 指标名称映射标准字典；数值清理单位文本、箭头、空格和全角符号后保存为规范字符串。
- 单位不一致时不静默换算，必须人工确认；本轮不支持跨单位换算和自定义指标。
- OCR 确认页允许刷新后恢复暂存结果，也允许放弃本次暂存报告并保持原档案不变。
- 既有档案补传、补传确认/取消和档案删除必须进行附件版本校验；并发冲突返回 409，不允许旧页面覆盖较新的解析结果。直接新建 OCR 档案仍使用普通 `parsed → confirmed` 流程。

### 3.4 机构管理员后台

- 每个机构最多一个机构管理员，每个机构管理员只绑定一个机构。
- 机构运营总览展示资料完整度、相册、套餐、本机构已确认档案和最近档案。
- 可修改所属机构资料，增删改查本机构套餐和相册。
- 可只读查看来源为本机构、状态为 `confirmed` 的档案、标准化指标、异常和趋势。
- 不返回邮箱、手机号、上传人或原始报告，不允许修改、补录、删除档案或把机构数据交给 AI。
- 查询只写不含指标值的结构化运行日志，不建立数据库审计页面。

### 3.5 系统管理员后台

- 系统总览显示角色分布、机构状态、邀请码状态、待审评论和档案统计。
- 对机构执行新增、编辑、软停用和恢复；对套餐执行增改和软停用。
- 签发、撤销和重新生成机构邀请码。
- 用户管理显示角色与绑定机构，并通过专用操作撤销机构管理员。
- 通过 admin 专属页面和接口监管全部档案，审核评论。

### 3.6 邀请码

- 邀请码单次使用、永不过期，每机构最多一个当前邀请码。
- 数据库只保存哈希；明文仅在签发时展示一次，不集成邮件或短信。
- 重新生成立即使旧码失效；消费邀请码与创建机构管理员在同一事务内完成。
- 撤销管理员后账号降级为普通用户，个人历史数据保留；重新任命必须签发新邀请码。

### 3.7 机构相册与生命周期

- 每机构最多 8 张 JPEG、PNG 或 WebP 图片，每张不超过 5 MB。
- 支持拖拽排序，排序第一张自动作为封面。
- 服务端实际解码图片、修正方向并清除 EXIF。
- 机构和套餐删除均为软停用；停用机构时撤销当前机构管理员和未使用邀请码，历史档案、套餐、评论和图片继续保留。

### 3.8 亲友、评论与 AI

- 普通用户继续管理亲友关系和授权，被添加方控制授权状态。
- 用户上传过对应机构档案后才允许评论；系统管理员审核公开状态。
- AI 继续执行科普、人工客服和急症分流，不诊断、不推荐处方药。
- 两类管理员后台不显示健康 AI，AI 不接受管理员提交的档案 ID。
- AI 侧栏初始不加载、不展示档案列表；只有用户主动点击“引用档案”或问题确实需要个人报告时，才显示内联选择器。
- 普通问题不发送档案。每次引用档案均需重新选择并明确同意，选择和同意不得自动延续到后续无关消息。
- AI 采用 SSE 流式交互，用户消息立即显示，并提供状态提示、取消、失败保留和重试；流事件固定为 `meta/status/delta/action/done/error`。
- 档案分析只接受本人或已授权亲友、状态为 `confirmed` 且至少有一项指标的档案；同一请求中的档案必须属于同一所有者，不设置用户可见数量上限。
- 多档趋势的排序、首次/最新值、极值、异常次数、缺失和数值变化由服务端确定性计算；同日多记录、非数值或不可比较情况必须明确标注，不让模型自行补算。
- 聊天和分析结果不写服务端数据库；当前标签页可以保存界面会话，敏感档案上下文不得自动带入其他轮次。

## 4. 技术方案

| 层级 | 选型 |
|---|---|
| 前端 | Vue 3、Vite 6、Vue Router、Pinia、Element Plus、ECharts 6 |
| 后端 | Flask、Flask-SQLAlchemy、Flask-JWT-Extended |
| 数据库 | 本地 SQLite，schema 版本使用 `PRAGMA user_version` |
| 文件 | 本地存储；机构图片公开，健康报告通过鉴权接口读取 |
| OCR | 本地 Mock 或华为云 OCR |
| AI | DeepSeek V4 Flash、SSE 流式输出、本地 FAQ、安全分流与确定性趋势事实 |
| 测试 | Pytest；前端补充 Vitest 与 Vue Test Utils |

## 5. 数据与权限原则

- `users.role` 取值为 `user/institution_admin/admin`。
- `managed_institution_id` 唯一且可空；机构管理员必须绑定机构，其他角色不能绑定。
- 机构健康数据使用独立只读接口和脱敏 DTO，不复用用户档案编辑接口。
- 普通用户、机构管理员和系统管理员分别使用个人、机构和 admin 档案接口。
- 角色变更后后端逐请求查库立即生效；前端在重新导航、刷新令牌或刷新页面时重新获取本人信息并重建导航。
- AI、邀请码、图片和报告文件均遵循最小权限与服务端校验。
- 普通用户和亲友代理均不能修改既有档案归属人；需要纠错时由系统管理员通过专属监管接口处理。
- production 默认仅监听本机地址；回环启动脚本自动生成随机 JWT 密钥。本机演示可使用受限演示账号，对外监听必须配置非演示管理员密码并保持强 JWT 密钥。

## 6. SQLite schema v2

系统继续使用当前 SQLite，不更换数据库。现有数据通过 schema 迁移保留：

- 新增机构管理员绑定、邀请码、相册和启用状态结构。
- 旧 `logo_url` 如有值，迁移为相册排序第一张。
- 迁移保留用户、机构、套餐、档案、指标、评论、主键和密码哈希。
- 非空旧库未迁移时拒绝启动，避免 `db.create_all()` 形成半升级库。
- 升级采用临时库重建、校验、备份和原子替换。

## 7. 非功能需求

- 支持 Windows PowerShell 本地启动，Python 3.10+、Node.js 20+。
- 密码和邀请码只保存哈希；密钥只来自被 Git 忽略的环境变量。
- 健康报告不能通过公开上传路径读取。
- 三工作台支持桌面和移动端，具备 403、404、空状态和加载反馈。
- 桌面端 AI 侧栏不得挤压导航文字，主页面按比例缩放且最低比例为 0.7；空间不足时切换为具备焦点管理的遮罩对话框。
- AI 请求应先返回 `meta/status` 状态，连接、读取和总截止默认分别为 5/30/60 秒；失败不能无限加载。
- 数据模型、接口、测试和文档必须同步更新。

## 8. 当前实现与验收结论

- 公开门户、独立认证页和三角色工作台均已落地，登录后按实时角色跳转。
- 三套角色路由与接口已隔离；后端逐请求查库校验角色与机构绑定。
- 邀请码、机构/套餐软停用、机构相册、机构健康数据只读投影、可选档案来源、指标值规范化和报告鉴权均已实现。
- AI 流式对话、按需选档、同所有者多档智能分析、确定性趋势、取消/重试和隐私同意重置已实现。
- OCR 新建档案与既有档案暂存录入、恢复/放弃、附件版本 CAS 和文件补偿清理已实现；展示编号已统一为 `health+数字`。
- 正式 SQLite 已迁移到 `PRAGMA user_version=2`，迁移前备份保留在 `backend/instance/`；完整性和外键检查通过。
- 迁移后保留 2 个用户、3 家机构、15 个套餐和 12 份档案。
- 后端 117 项自动化测试和前端 18 个文件共 97 项 Vitest 测试通过；前端生产构建通过，生产依赖审计为 0 个漏洞。
- GitHub Actions 在 `main` push/PR 上使用 Python 3.12 与 Node.js 20 执行后端测试、前端测试和生产构建；2026-07-13 发布 SHA `77d3443b` 的 CI 已通过。
- 当前不包含跨单位自动换算、自定义指标、数据库审计页面或 PDF 重新制作。
