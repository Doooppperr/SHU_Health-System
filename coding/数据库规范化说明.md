# 数据库规范化说明

本项目的数据库按实体表、字典表、关系表拆分，目标范式为 3NF，关键业务表通过候选键约束强化到 BCNF。当前结构已迁移并应用到华为云 GaussDB。

## 关系模式

| 表 | 主键 | 候选键/唯一约束 | 说明 |
|---|---|---|---|
| `users` | `id` | `username`、`email` | 用户账号信息，用户属性只依赖用户键 |
| `institutions` | `id` | `(name, branch_name)` | 体检机构分院，一个分院只保存一次 |
| `packages` | `id` | `(institution_id, name)` | 套餐从属于机构，套餐详情只依赖机构内套餐名 |
| `indicator_categories` | `id` | `name` | 指标分类字典 |
| `indicator_dicts` | `id` | `code`、`(category_id, name)` | 指标字典，参考范围、单位、临床意义只依赖指标键 |
| `health_records` | `id` | 无业务唯一键 | 体检档案事实表，关联归属人、上传人、机构和套餐 |
| `health_indicators` | `id` | `(record_id, indicator_dict_id)` | 档案和指标的多值关系，避免同一档案重复录入同一指标 |
| `friend_relations` | `id` | `(user_id, friend_user_id)` | 用户和亲友的授权关系 |
| `comments` | `id` | 无业务唯一键 | 评论事实表，允许同一用户对同一机构多次评论 |

## 3NF 判断

1. 每张表字段都是原子值，满足 1NF。
2. 复合唯一键表中，非键属性依赖完整候选键：
   - `packages` 的 `price`、`focus_area` 等依赖 `(institution_id, name)`，不存在只依赖 `institution_id` 的部分依赖。
   - `health_indicators` 的 `value`、`is_abnormal` 依赖 `(record_id, indicator_dict_id)`。
3. 非键属性不依赖其他非键属性：
   - 机构信息不重复存入档案表，只通过 `institution_id` 引用。
   - 套餐信息不重复存入档案表，只通过 `package_id` 引用。
   - 指标单位、参考范围、临床意义不重复存入体检结果表，只通过 `indicator_dict_id` 引用。
   - 用户信息不重复存入评论、档案、亲友关系表，只通过用户外键引用。

因此整体满足 3NF。

## BCNF 强化

BCNF 要求每个非平凡函数依赖的决定因素都是超键。项目通过以下唯一约束表达业务候选键：

- `users.username -> users.*`
- `users.email -> users.*`
- `institutions.(name, branch_name) -> institutions.*`
- `packages.(institution_id, name) -> packages.*`
- `indicator_categories.name -> indicator_categories.*`
- `indicator_dicts.code -> indicator_dicts.*`
- `indicator_dicts.(category_id, name) -> indicator_dicts.*`
- `health_indicators.(record_id, indicator_dict_id) -> health_indicators.*`
- `friend_relations.(user_id, friend_user_id) -> friend_relations.*`

这些决定因素都被声明为主键或唯一键，核心字典表和关系表满足 BCNF。`health_records` 与 `comments` 是事件/事实表，没有稳定业务唯一键，因此使用代理主键 `id` 保证实体标识。

## 并发能力

当前后端使用 Flask-SQLAlchemy 的请求级 Session，生产启动脚本使用 Waitress 多线程服务，数据库侧使用 GaussDB/openGauss 的事务与 MVCC 能力。应用配置中增加了连接池参数：

- `DB_POOL_SIZE=10`
- `DB_MAX_OVERFLOW=20`
- `DB_POOL_TIMEOUT=30`
- `DB_POOL_RECYCLE=1800`

因此多用户同时登录、录入档案、提交评论时，不会共用同一个全局数据库连接；并发写入冲突由数据库事务、唯一约束、外键和触发器共同兜底。

## 断言式约束与触发器

标准 SQL 的 `CREATE ASSERTION` 在 GaussDB/openGauss 实际工程中不作为常规可用能力使用。本项目采用更可落地的数据库级 `CHECK` 约束和触发器实现断言式规则：

- `CHECK` 约束：限制用户角色、档案状态、评论评分、套餐价格、指标来源、非空文本、参考范围上下界等。
- 触发器 `trg_health_records_package_match`：保证档案中的 `package_id` 必须属于同一 `institution_id`，如果只传套餐则自动补齐机构。
- 触发器 `trg_comments_require_uploaded_record`：保证用户对机构评论前，必须已上传该机构体检档案。
- 触发器 `trg_health_indicators_set_abnormal`：写入或更新指标值时，根据指标字典参考范围自动维护 `is_abnormal`。

模型层的新建库规则保存在 SQLAlchemy 模型中；现有云库可通过 `backend/scripts/apply_gaussdb_rules.py` 补齐。

现有云库补齐这些规则需要管理员账号执行 DDL；应用账号 `health_app` 保留最小业务权限，不授予表结构变更权限。

当前云端 GaussDB 已完成规则落库并验证：

```text
CHECK 约束：22 个
触发器：3 个
触发器清单：
- trg_health_records_package_match
- trg_comments_require_uploaded_record
- trg_health_indicators_set_abnormal
```

验证方式为事务内插入非法数据并回滚：评论评分越界被 `CHECK` 拦截、档案套餐机构不匹配被触发器拦截、指标异常标记由触发器自动维护。

comments: 1

后续如果新增业务表，应继续遵循：

1. 字典/主数据独立成表，业务事实表只存外键。
2. 可自然唯一识别实体的字段组合必须加唯一约束。
3. 不在事实表中重复保存可由外键推导出的机构名、套餐名、指标单位、参考范围等描述字段。
